# Output formats

Every report can be produced in one of four formats. HTML is the default; PDF and CSV are instant follow-ups after any HTML report; PPTX is for client-facing executive decks.

## HTML (default)

- Primary deliverable for every report
- Interactive (Chart.js charts, hover tooltips)
- Renders in any modern browser
- Generated by injecting the summary JSON into `templates/report-shell.html`
- Supports `Print → Save as PDF` from the browser for a quick PDF (uses `@media print` block in the main stylesheet)

## PDF (native)

- Generated by `scripts/build_pdf.py` using WeasyPrint
- JS-free — charts are pre-rendered as SVG by matplotlib via `scripts/chart_renderer.py`
- Uses `scripts/pdf_styles.css` (a JS-free variant of the main stylesheet)
- Fonts bundled in `assets/fonts/` — no network dependency at render time
- Letter-size, margined for printing
- Native vector PDF — crisp at any zoom level, prints perfectly

### When to offer PDF

After delivering any HTML report, offer this as a one-line follow-up:

> Want this as a PDF to send to your team?

Only generate unprompted if the user's original request explicitly said "PDF."

### How to build

```python
from build_pdf import build_pdf_from_summary
build_pdf_from_summary(summary, '/mnt/user-data/outputs/report.pdf')
```

Takes the same `summary` dict as the HTML template. No translation needed.

## CSV (raw data)

- Generated by `scripts/build_csv.py`
- One row per entry, columns for every UTM field, click ID, contact field, and form response
- UTF-8 encoded, Excel- and Google-Sheets-compatible
- Column order optimized for attribution analysis: identity first, UTM fields second, click IDs third, page context, form responses, technical details last

### When to offer CSV

Same pattern — one-line follow-up after any HTML report:

> Want the underlying leads as a CSV?

Appropriate when the user might want to slice data differently, run it through their own analytics, or hand it to a sales rep to work.

### How to build

```python
from build_csv import export_entries_to_csv
export_entries_to_csv(entries_list, '/mnt/user-data/outputs/leads.csv')
```

Takes the raw entries list (same one used to compute the report).

## PPTX (native PowerPoint)

- Generated by `scripts/build_pptx.py` using `python-pptx`
- 16:9 widescreen (13.33" × 7.5"), one slide per section
- Two-tone cards rendered as stacked rectangles (tinted top + white bottom, hairline border)
- Italic `*word*` accents rendered as italic runs in brand primary color
- Charts rendered as high-DPI PNG images (via `scripts/chart_renderer_png.py`)
- Headers and footers on every slide (company name, skill version, date, slide N/total)
- Native-editable: text frames, shapes, and images all individually selectable and editable in PowerPoint/Keynote/Google Slides
- Fonts: Instrument Serif, Geist, Geist Mono — falls back to Georgia/Calibri/Consolas gracefully

### When to offer PPTX

One-line follow-up after any HTML report:

> Want this as a PowerPoint deck to send to your team?

Appropriate when the user will present to executives, paste into an existing master deck, or hand off to someone who needs to edit copy before sharing.

### How to build

```python
from build_pptx import build_pptx_from_summary
build_pptx_from_summary(summary, '/mnt/user-data/outputs/report.pptx')
```

Takes the same `summary` dict as the HTML template. All 10 section types supported.

## Multi-format delivery

If a user asks for multiple formats in one request (e.g., "give me the monthly as HTML and PDF"), run the HTML first, then the PDF from the same `summary` dict — no extra MCP calls needed. The cached entries + computed summary are reused.

Order of delivery:
1. Generate HTML first and `present_files` it
2. Generate PDF second (same summary dict, no recompute)
3. Call `present_files` again with both HTML and PDF paths

CSV can slot in anywhere and reuses the cached entries list.
